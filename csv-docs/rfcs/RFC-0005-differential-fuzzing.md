# RFC-0005: Differential Fuzzing Framework

**Status:** Proposed
**Author:** Parwana Core Team
**Date:** 2026-05-21
**Version:** 1.0.0

## Abstract

This RFC establishes a differential fuzzing framework for the Parwana's canonical serialization and hashing. It compares multiple implementations (Rust, TypeScript) to detect serialization inconsistencies that could lead to verification failures.

## Motivation

The current test suite includes property-based tests but lacks:

1. **Cross-implementation fuzzing** — No comparison between Rust and TypeScript implementations
2. **Differential fuzzing** — No framework for feeding the same inputs to multiple implementations
3. **Fuzz corpus management** — No structured corpus for regression testing
4. **Coverage-guided fuzzing** — No integration with libFuzzer or AFL++

## Design

### 1. Fuzz Target Architecture

```
fuzz/
├── Cargo.toml
├── fuzz_targets/
│   ├── canonical_cbor_roundtrip.rs    # Serialize → Deserialize roundtrip
│   ├── canonical_cbor_differential.rs # Rust vs TypeScript serialization
│   ├── proof_bundle_decode.rs         # ProofBundle deserialization
│   ├── sanad_envelope_decode.rs       # SanadEnvelope deserialization
│   ├── merkle_proof_verify.rs         # Merkle proof verification
│   └── domain_hash_consistency.rs     # Domain hash consistency
├── corpus/
│   ├── canonical_cbor_roundtrip/
│   ├── canonical_cbor_differential/
│   ├── proof_bundle_decode/
│   └── ...
└── README.md
```

### 2. Differential Fuzzing Protocol

For each fuzz target:

1. Generate random input
2. Serialize in Rust implementation
3. Serialize in TypeScript implementation
4. Compare byte-for-byte
5. If different, report as failure

### 3. Corpus Management

The corpus is organized by fuzz target:

```
corpus/
└── <target_name>/
    ├── 00000001-cbor-proof-bundle.cbor
    ├── 00000002-cbor-sanad-envelope.cbor
    └── ...
```

Corpus entries are:
- Generated by the fuzzer during execution
- Validated against the golden corpus
- Committed to version control for reproducibility

### 4. Integration with CI

```yaml
# .github/workflows/fuzz.yml
name: Fuzzing
on:
  push:
    paths:
      - csv-core/src/canonical.rs
      - csv-core/src/hash.rs
      - csv-core/src/merkle.rs
  schedule:
    - cron: '0 0 * * 1'  # Weekly
jobs:
  fuzz:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run fuzz targets
        run: cd fuzz && cargo fuzz run canonical_cbor_differential -- -max_total_time=3600
```

## Implementation

### New Crate: `fuzz/`

- `Cargo.toml` with `cargo-fuzz` dev-dependency
- Fuzz targets for each critical path
- Corpus directory with initial entries from golden corpus

### Changes to `csv-core/Cargo.toml`

- Add `[[bin]]` target for `generate_corpus` binary
- Add `dev-dependencies` for fuzzing infrastructure

### New Tests

- `csv-core/tests/properties/differential_serialization.rs` — Rust vs TypeScript comparison
- `csv-core/tests/properties/fuzz_regression.rs` — Corpus-based regression tests

## Security Impact

- **Detects serialization inconsistencies** — Cross-implementation comparison catches subtle bugs
- **Regression prevention** — Corpus-based testing catches regressions early
- **Coverage-guided discovery** — libFuzzer finds edge cases humans miss
